# ADR-001 - Adoption de l’architecture hexagonale avec DDD

## Statut
Accepté

## Date
2026-03-26

## Contexte
Le projet est un portail web de mutuelle permettant à un utilisateur de :

- gérer son compte (authentification, profil, 2FA)
- soumettre des demandes de remboursement avec pièces jointes
- suivre ses remboursements et virements
- consulter et télécharger ses documents
- gérer son contrat
- contacter le support
- recevoir des notifications (emails via workers)

Le système contient des règles métier non triviales :
- cycle de vie d’un remboursement
- validation des demandes
- gestion des documents
- traitements asynchrones (workers)
- intégration avec des services externes (email, stockage, etc.)

L’application doit :
- isoler la logique métier des aspects techniques
- être testable
- pouvoir évoluer sans couplage fort aux frameworks
- refléter le langage métier (mutuelle)

## Décision
Nous adoptons :

- **l’architecture hexagonale (Ports & Adapters)**
- **Domain-Driven Design (DDD)** pour la modélisation métier

### Structure de l’application

- **Domaine**
    - entités métier
    - value objects
    - services métier
    - règles de gestion

- **Application**
    - cas d’usage
    - orchestration
    - ports d’entrée et de sortie

- **Infrastructure**
    - contrôleurs (REST)
    - persistance (DB)
    - stockage de fichiers
    - envoi d’emails
    - workers / messaging

### Règle de dépendance

- Infrastructure → Application → Domaine
- Le domaine ne dépend d’aucune couche technique

### Ports et adapters

Toutes les dépendances externes passent par des ports :
- repository
- service de notification
- stockage de documents
- système de messaging

Les adapters implémentent ces ports.

### DDD

Le domaine est structuré autour des concepts métier :

- remboursement
- contrat
- document
- utilisateur
- notification

Des sous-domaines peuvent émerger :
- Authentification
- Remboursement
- Documents
- Contrat
- Support

## Alternatives considérées

### 1. Architecture en couches classique (Controller → Service → Repository)
Rejetée car :
- logique métier dispersée
- couplage fort aux frameworks
- difficile à faire évoluer proprement

---

### 2. Architecture CRUD simple
Rejetée car :
- le domaine métier est non trivial
- workflows (remboursement, notifications) mal modélisés

---

### 3. Microservices dès le départ
Rejetée car :
- complexité opérationnelle élevée
- frontières métier encore instables
- surdimensionné pour le besoin actuel

Une évolution future reste possible.

---

### 4. DDD sans architecture hexagonale
Rejetée car :
- risque de dérive du domaine vers des dépendances techniques
- perte progressive de la séparation métier / technique

## Conséquences

### Positives
- séparation claire entre métier et technique
- meilleure testabilité
- code aligné avec le métier
- facilité d’évolution
- gestion propre des intégrations externes
- adapté aux traitements asynchrones (workers)

### Négatives
- complexité initiale plus élevée
- nécessite une discipline d’équipe
- courbe d’apprentissage
- risque d’over-engineering si mal appliqué

## Portée
Cette décision s’applique à l’ensemble du backend.

Le DDD sera appliqué de manière pragmatique :
- pas de sur-modélisation inutile
- priorité à la clarté et à la valeur métier
